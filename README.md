# Go-Honeypot: Модульный симулятор CDN-ориджина

Высокопроизводительный бэкенд-заглушка на Go для интеграции с цепочкой Nginx -> Xray (vlessenc/xhttp). 
Предназначен для защиты прокси-узлов от активного пробинга и поведенческого анализа систем DPI.

## Ключевые особенности
- **Zero Disk I/O:** Все медиа-сегменты (`.ts`) загружаются в RAM при старте приложения. Полное отсутствие задержек чтения.
- **Умная сепарация:** Поддержка отдачи как случайных чанков, так и жестко структурированных по `seqKey` (эмуляция реального плеера).
- **Параллельный прогрев кэша:** Встроенный фоновый воркер для автоматической генерации «белого» трафика через прокси-туннели.
- **Модульная архитектура:** Возможность добавления новых типов маскировки (например, симуляция API или обычных HTML-страниц) через реализацию интерфейса `Module`.

## Настройка и запуск
1. Разместите `.ts` файлы в директории, указанной в `config.yaml` (`modules.videostream.chunks_dir`).
2. Запустите приложение:
```bash
go build -o honeypot cmd/honeypot/main.go
./honeypot -config config.yaml
```

## Полное описание конфигурации (`config.yaml`)

Файл `config.yaml` управляет поведением ядра и всех подключенных модулей. Ниже приведено описание каждого параметра: зачем он нужен, как работает и в каких случаях его стоит включать или отключать.

### Блок `server` (Сетевые настройки)
Глобальные настройки HTTP-сервера.
*   **`listen`** (`строка`): IP-адрес и порт, на которых приложение будет слушать входящий трафик.
    *   *Когда нужно:* Всегда.
    *   *Пример:* `"127.0.0.1:8081"` — слушать только локальные подключения (рекомендуется для связки из связки с Nginx/Xray). `"0.0.0.0:8081"` — открыть порт в интернет.
*   **`read_timeout`** (`строка`): Максимальное время (например, `30s`) ожидания чтения запроса от клиента.
    *   *Для чего:* Защита от атак (таких как Slowloris), когда клиент специально медленно отправляет байты, истощая пулы соединений.
*   **`write_timeout`** (`строка`): Максимальное время (например, `60s`) ожидания отправки ответа.
    *   *Для чего:* Если клиент качает файл на скорости 1 KB/s, соединение будет принудительно разорвано через 60 секунд.
*   **`max_header_bytes`** (`число`): Максимально допустимый размер HTTP-заголовков в байтах (например, `1048576` — это 1 Мегабайт).
    *   *Для чего:* Защищает оперативную память от переполнения гигантскими заголовками. 1MB хватает с головой.

### Блок `logging` (Журналирование)
Отвечает за формат вывода диагностических сообщений приложения в консоль (stdout).
*   **`level`** (`строка`): Уровень логирования. Варианты: `debug` (все детали), `info` (основное), `warn` (предупреждения), `error` (только ошибки).
    *   *Когда нужен `debug`:* На этапе настройки или при поиске проблем маршрутизации.
    *   *Когда нужен `info/error`:* На постоянно работающем продакшен-сервере, чтобы не забивать логи и не тратить диск.
*   **`format`** (`строка`): Формат представления данных: `text` (читаемый текст) или `json` (структурированный JSON).
    *   *Когда нужен `json`:* Если у вас настроен сбор логов (Elasticsearch, Filebeat, Grafana Loki), который автоматически парсит JSON-поля.

### `enabled_modules` (Список активных модулей)
Массив строк. Здесь перечисляются имена модулей, которые диспетчеру разрешено запустить. 
*   *Особенность:* Даже если модуль детально настроен в секции `modules`, но его имени нет в `enabled_modules` — он **не загрузится** и не заберет на себя системные ресурсы. Это удобно для быстрого включения/отключения блоков (например, можно легко отключить прогрев кэша, закомментировав одну строчку).

### Подраздел `modules.videostream` (Модуль эмуляции контента)
Этот блок отвечает за подмену ответов (имитацию видеостриминга для DPI систем).
*   **`path`** (`строка`): URI маршут (путь), по которому модуль ожидает запросы (например, `"/media/stream/part"`). 
*   **`chunks_dir`** (`строка`): Путь на сервере к директории (например, `"/var/www/html/CHUNKS"`), в которой лежат `.ts` (MPEG-TS) файлы — частицы видео на замену.
*   **`preload_to_ram`** (`boolean`):
    *   `true`: При старте программы **все** файлы из папки `chunks_dir` полностью считываются в оперативную память. Хонипот отдаёт файлы из RAM за сотые доли миллисекунд (Zero Disk I/O). Запросы блокировок систем глубокого анализа трафика будут сглаживаться идеальными таймингами отдачи, не нагружая диск сервера. *Обязательно используйте в продакшене при достаточном количестве ОЗУ.*
    *   `false`: Читает чанки напрямую с диска только в момент запроса.
*   **`delivery_mode`** (`строка`): Режим отдачи видеофрагментов.
    *   `sequential`: Ищет указанный ключ (см. `seq_key`) в URL, чтобы отдать конкретный файл (например `1.ts`, `2.ts`). Симулирует реальные плееры. Наиболее правдоподобный режим для обхода продвинутых файрволов (GFW).
    *   `random`: По каждому запросу отдаёт случайный чанк. Удобно, когда структура URL неизвестна.
*   **`seq_key`** (`строка`): Название GET-параметра при режиме `sequential` (Пример: `"id"` для запросов вида `?id=12`).
*   **`fallback_mode`** (`строка`): Что делать, если включен режим `sequential`, но пользователь не передал GET-параметр или запрошенного файла не нашлось.
    *   *Значение `random`:* Отдаст любой другой кусок видео вместо ошибки HTTP 404 (чтобы не привлекать подозрение цензоров частыми 404-ошибками, т.к. "белые" CDN редко отдают пустоты).
*   **`headers`** (`словарь`): Ключи и значения кастомных HTTP-заголовков.
    *   *Применяемость:* Критически важно для маскировки. Нужно указывать `Server: nginx/1.24.0`, чтобы сымитировать, что бэкендом выступает именно NGINX, и `Content-Type: video/mp2t` (MIME-тип TS файлов).

### Подраздел `modules.cache_warmer` (Модуль прогрева и генерации шума)
Модуль создает дополнительный «белый» (на первый взгляд легитимный) исходящий трафик с самого сервера. Это помогает обмануть поведенческие анализаторы провайдера, создавая картину того, что ваш сервер действительно общается с внешними CDN.
*   **`enabled_worker`** (`boolean`): Локальный рубильник модуля. `true` позволяет ему работать на фоне. `false` выключает фоновый процесс на уровне горутин.
*   **`interval_seconds`** (`число`): Как часто (в секундах) запускать "волну" запросов.
    *   *Настройка:* Чем меньше, тем больше трафика. Оптимально 30-60 сек.
*   **`concurrency_limit`** (`число`): Максимальное количество параллельных запросов за один `interval_seconds`.
    *   *Для чего:* Защищает от само-DDoS-атак. Если у вас в `targets` 100 URL, а `concurrency_limit` стоит \`2\`, программа будет скачивать их строго по 2 файла за раз, не забивая интернет-канал.
*   **`local_proxy`** (`строка`): URL локального прокси-сервера (чаще всего это SOCKS5 порт от Xray/V2ray, или WARP). Например: `"socks5://127.0.0.1:40000"`.
    *   *Когда нужно:* Если вы хотите, чтобы ваш «прогревочный» фейковый трафик маршрутизировался через те же зашифрованные прокси-туннели, через которые ходят реальные клиенты, зашумляя канал. Во всех остальных случаях оставьте поле пустым `""` (пойдет напрямую).
*   **`targets`** (`массив строк`): Список интернет-адресов. Именно на эти URL воркер будет периодически отправлять `GET` запросы.
